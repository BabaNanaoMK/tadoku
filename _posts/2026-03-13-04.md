## Agent Manager 画面と Conversation 画面の関係

この図は、Antigravity における **Agent Manager** と **Conversation** の役割分担を整理したものです。
ポイントは、**Conversation は個別の作業単位であり、それらを束ねて管理する上位画面が Agent Manager である**、ということです。

---

## まず結論

* **Agent Manager = 全体管理の画面**
* **Conversation = 1件ごとの作業スレッド**

この2つは対立する概念ではなく、**親子関係**にあります。
つまり、Conversation は Agent Manager の中で扱われる構成要素のひとつです。

---

## 図の見方

図の上部にある大きな枠が **Agent Manager** です。
その中に、複数の **Conversation** が配置されています。

たとえば図では、以下のような会話単位が並んでいます。

* Conversation A（コード生成）
* Conversation B（データ分析）
* Conversation C（リサーチ）

これらはそれぞれ別々の依頼や作業を表しています。
つまり Conversation は、Slack で言えば1本のスレッド、Issue管理で言えば1件のタスク、開発現場で言えば1トピックの作業単位に近いものです。

一方で Agent Manager では、これら複数の Conversation だけでなく、右側にあるような周辺要素もまとめて扱います。

* **Artifacts（成果物）**
* **Knowledge Items（知識）**
* **Terminal / Browser / Pane（操作画面）**

このため Agent Manager は、単なるチャット一覧ではなく、**作業全体の進行・資産・操作面を束ねる管理ハブ**として理解するとわかりやすいです。

---

## Conversation は何をする画面か

Conversation 画面は、**個別案件を実行する場**です。
ここでユーザーはエージェントに対して依頼を書き、進行を追い、必要に応じて追加指示や確認を行います。

図の中央の説明を言い換えると、Conversation の役割は次の3点に集約できます。

* **1件の依頼を扱う**
* **指示を書く**
* **進行を見る**

つまり Conversation は、実務的には「作業スレッド」そのものです。
コード生成、調査、文章作成、修正依頼などを、トピックごとに切り分けて進める場所だと考えると自然です。

---

## Agent Manager は何をする画面か

Agent Manager は、**複数の作業を俯瞰して扱う管理画面**です。
図では「司令室」と表現されていますが、IT寄りに言えば、**ワークスペース全体のオーケストレーション層**と見てもよいです。

主な役割は以下です。

* **全体を見る**
* **複数タスクを扱う**
* **成果物や知識も管理する**

Conversation 単体では「いまこの依頼で何をしているか」が中心になりますが、Agent Manager では「他の依頼はどうなっているか」「生成済みの成果物は何か」「参照知識は何か」といった、より広い視点での管理ができます。

この違いがあるので、Conversation は実行単位、Agent Manager は管理単位、と整理すると混乱しにくくなります。

---

## 両者の関係をIT的に言い換えると

この図の内容は、次のように置き換えると理解しやすいです。

* **Agent Manager**
  ワークスペース / 管理コンソール / プロジェクト全体ビュー
* **Conversation**
  個別チケット / 個別スレッド / 個別タスク

つまり Agent Manager は「プロジェクト全体を見る場所」、Conversation は「その中で1件ずつ仕事を進める場所」です。

---

## 操作の流れ

図の下部には、基本的な操作フローも示されています。

1. **Agent Manager で会話を作る**
2. **Conversation で依頼する**
3. **作業の進行を見る**
4. **成果物を確認する**

この流れからわかるのは、**作業の開始点は Agent Manager だが、実際の実行単位は Conversation である**ということです。

実際の運用では、たとえば次のように使い分けることになります。

* 新しい調査テーマごとに Conversation を分ける
* コード修正と仕様整理を別 Conversation に分ける
* 生成されたファイルやメモは Agent Manager 側で横断的に確認する

このように、**Conversation を細かく切り分け、Agent Manager で束ねて見る**という運用が基本になります。

---

## なぜ分けて考える必要があるのか

Antigravity を初めて触ると、Conversation がメイン画面に見えるので、「Conversation が全部なのでは」と感じやすいです。
ですが実際には、Conversation はあくまで **作業の実行単位** です。

複数の作業が並行し、成果物や知識も蓄積されると、必要になるのは個別スレッドではなく、**全体を見渡して整理する視点**です。
その役割を担うのが Agent Manager です。

この構造を理解しておくと、

* どこで新しい作業を切るべきか
* どこで成果物を見ればよいか
* どこで全体状況を把握すればよいか

がクリアになります。
ここが曖昧だと、1本の Conversation に全部を詰め込み、運用がぐちゃっとしがちです。設計をサボると作業がスライム化します。

---

## まとめ

この図が伝えたいことを一文でまとめると、次の通りです。

**Conversation は「作業単位」、Agent Manager は「それらを束ねる管理画面」** です。

したがって、Antigravity を使うときは、

* **個別作業は Conversation で進める**
* **全体管理は Agent Manager で行う**

という二層構造で捉えるのが実用的です。

---

## 図と一緒に添える短い説明文（要約版）

Antigravity では、**Conversation** は1件ごとの依頼や作業を扱うスレッドであり、**Agent Manager** はそれら複数の Conversation や成果物、知識、操作画面をまとめて管理する上位画面です。

つまり、Conversation は「実行単位」、Agent Manager は「管理単位」です。
新しい作業を切り、個別に進め、全体を俯瞰するために、この2つを分けて理解することが重要です。
