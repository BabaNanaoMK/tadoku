# Codex CLI のモデル、実務ではどう使い分ける？

Codex CLI を使っていると、モデル選択にいろいろ並んでいて、最初はちょっと面食らいます。
名前が似ていて、しかも「どれを使えばいいのか」が直感で分かりにくい。実に機械っぽい世界です。

結論から言うと、**実務ではまず GPT-5.4 を基本にする**のがいちばん分かりやすく、失敗しにくい運用です。OpenAI の公式ドキュメントでも、**Codex では most tasks は GPT-5.4 から始める**と案内されています。 ([OpenAI Developers][1])

## 結論：おすすめ運用はこの2段構え

実務でのおすすめ運用は、とても単純です。

### 普段は GPT-5.4

日常的な開発作業、新機能追加、バグ修正、設計相談、シェルや Python の自動化、ドキュメント整備まで含めて、**まずは GPT-5.4 で始める**のが基本です。GPT-5.4 は、コーディングだけでなく、推論、ツール利用、複数ステップの業務処理まで含めた「主力モデル」として案内されています。 ([OpenAI][2])

### コード寄りの重め作業は GPT-5.3-Codex も有力

複数ファイルをまたぐ改修、長めの実装、ツールを使いながらの調査や実行など、**かなり開発色の強いタスク**では GPT-5.3-Codex も有力です。公式には、GPT-5.3-Codex は **agentic coding tasks に最適化された高性能モデル**として紹介されています。 ([OpenAI Developers][3])

つまり、実務感覚ではこうなります。

* **迷ったら GPT-5.4**
* **ガチ開発モードなら GPT-5.3-Codex も試す**

この二択を軸にしておけば、だいたい十分です。モデル選択で悩みすぎると、本来の仕事より設定画面をいじる時間の方が増えるので、本末転倒の小さな喜劇になります。

## なぜ GPT-5.4 を基本にしてよいのか

GPT-5.4 は、OpenAI の公式説明では、**GPT-5.3-Codex の最先端のコーディング能力を取り込みつつ、より広い実務タスクに対応する主力モデル**です。Codex 向けの公式ページでも、「ほとんどの作業は GPT-5.4 から始める」とされています。 ([OpenAI][2])

つまり GPT-5.4 は、

* コードを書く
* 既存コードを読む
* 調査する
* 方針を考える
* 文章も整える
* ツールを使って進める

といった、**実務で混ざりがちな仕事をまとめて扱いやすい**のが強みです。
Kindle 本の制作、シェル自動化、ブログ整備、Git 操作、リポジトリ改修のように、「コードだけでは終わらない作業」が多い人ほど、GPT-5.4 の恩恵を受けやすいです。 ([OpenAI][4])

## GPT-5.3-Codex はどう位置づけるべきか

GPT-5.3-Codex は、OpenAI によると **“the most capable agentic coding model to date”** と紹介されている、非常に強力なコーディングモデルです。GPT-5.2-Codex のコーディング性能と GPT-5.2 の推論力を統合し、さらに高速化したとされています。 ([OpenAI][5])

このため、次のような場面では相性がよいです。

* 長時間かかる実装
* 複数ファイル修正
* ツール利用を含む開発タスク
* 既存コードベースの理解と改修
* 「同僚っぽく継続して作業してもらう」使い方

ただし今の公式の立て付けでは、**まずは GPT-5.4 を使い、必要に応じて GPT-5.3-Codex を選ぶ**のが自然です。 ([OpenAI][2])

## それ以外のモデルはどう考えるか

ここから先は、簡単な位置づけだけ押さえておけば十分です。

### GPT-5.2-Codex

旧世代のコーディング特化モデルです。公式の一覧では、**long-horizon な agentic coding tasks 向け**とされています。悪いモデルではありませんが、今から新規に主力として選ぶ優先度は高くありません。比較検証や互換性確認のために使う場面が中心です。 ([OpenAI Developers][6])

### GPT-5.2

通常系の GPT-5.2 です。もちろんコードも書けますが、Codex CLI の文脈では、今は GPT-5.4 の方が主力として自然です。 ([OpenAI][4])

### GPT-5.1-Codex-Max

5.1 系の中では、**長時間タスク向け**の位置づけです。けれども世代が古いため、特別な理由がなければ GPT-5.4 か GPT-5.3-Codex を優先した方が分かりやすいです。 ([OpenAI Developers][6])

### GPT-5.1-Codex-Mini

小さく、軽く、コスト効率を重視したモデルです。公式にも **smaller, more cost-effective, less-capable** とあります。単純な補助作業には向きますが、本格的な開発の主役にすると、賢さ不足で回り道になることがあります。 ([OpenAI Developers][6])

## 実務でのおすすめ運用パターン

実際には、次のように考えると迷いません。

### 1. 普段の作業

**GPT-5.4 を使う。**

機能追加、バグ修正、設計相談、README 更新、シェルスクリプト作成など、普段の仕事はこれで十分です。 ([OpenAI Developers][1])

### 2. 長めで重い開発タスク

**GPT-5.3-Codex を試す。**

大きめの改修や、複数ファイル編集、ツールをまたぐ作業では、こちらがしっくり来ることがあります。 ([OpenAI Developers][3])

### 3. 旧モデルは特殊事情があるときだけ

比較、互換性確認、古い運用の再現など、理由があるときだけ使えば十分です。 ([OpenAI Developers][6])

## まとめ

Codex CLI のモデル選択は、最初はややこしく見えます。
でも、実務ではそこまで複雑に考えなくて大丈夫です。

**おすすめの結論はこれです。**

* **基本は GPT-5.4**
* **重めのコード作業では GPT-5.3-Codex も有力**
* **それ以外は補助的・旧世代と考えてよい**

この運用なら、モデル選択に振り回されず、本来の作業に集中できます。
機械の選択肢は多いですが、人間の時間は有限です。そこがこの話のいちばん大事なところです。
