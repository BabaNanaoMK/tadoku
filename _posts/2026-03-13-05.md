# Antigravity の「Context access」とは何か  

## Kindle本作成プロジェクトを例に、初心者向けにわかりやすく整理する

Antigravity のエージェント設定に **Context access** という項目があります。  
これが何を意味するのか、最初はかなり分かりにくいと思います。

ひとことで言えば、Context access は、

> **エージェントが作業をうまく進めるために、背景資料や前提知識を読める仕組み**

です。

Editor access が「ファイルを編集する力」  
Terminal access が「コマンドを実行する力」  
Browser access が「画面を開いて確認する力」  
だとすると、

**Context access は「事情を知る力」** です。

---

# Context access は何のためにあるのか

たとえば、エージェントにこんな依頼をするとします。

> この Kindle 本の原稿を、今の文体に合わせて整理してほしい

このとき、エージェントがただ文章を書くだけなら、一般論っぽいそれらしい文章は出せます。  
でも、それだけでは困ることがあります。

たとえば、

- この本の読者は誰か
- どんな語り口にしたいか
- 何を避けたいか
- 過去にどういう方針で書いてきたか
- どんな章構成を考えているか

こうした背景を知らないまま書くと、**それっぽいけれどズレた文章** になりやすいのです。

そこで必要になるのが Context access です。

---

# たとえるとどういうものか

Context access は、  
**新しく入った優秀な外注さんに「まずこの資料棚を見てから仕事してね」と渡す権限**  
だと思うと分かりやすいです。

仕事を頼むときに必要なのは、だいたい次の3つです。

1. 作業する道具  
2. 作業内容の指示  
3. 背景資料

この3番目が Context access です。

つまり Context access は、

> **この仕事を、空気を読んで進めるための資料置き場**

と考えるとよいでしょう。

---

# Context access でアクセスされるものの例

## 1. 基本方針ファイル

代表例が **`GEMINI.md`** です。

これは、プロジェクト全体の基本方針や前提を毎回読み込ませるためのファイルです。

たとえば、以下のようなことを書きます。

- このプロジェクトの目的
- 想定読者
- 文体
- 避けたい表現
- 重視すること
- 禁止事項

つまり `GEMINI.md` は、  
**毎回読む「現場マニュアル」** のようなものです。

---

## 2. Skills

**Skills** も広い意味では Context access に関係します。  
ただし、`GEMINI.md` とは少し役割が違います。

- `GEMINI.md` は毎回読む基本前提
- `SKILL.md` は必要なときだけ呼び出す専門手順

たとえば、

- 目次改善 Skill
- 文体調整 Skill
- Kindle向け推敲 Skill
- 具体例追加 Skill

のように、特定の仕事だけに使う専門ノートが Skills です。

つまり Skills は、

> **必要になったときだけ開く専門家の手順書**

と考えるとわかりやすいです。

---

## 3. プロジェクト内の説明ファイル

Context access の対象になりやすいものとして、次のようなファイルもあります。

- `README.md`
- `docs/` 配下の仕様書
- 設計メモ
- 命名規則
- 作業引き継ぎメモ
- フォルダ構成の説明

これらは、エージェントが作業前に参照すると役立つ資料です。

---

## 4. 外部ドキュメントや接続された知識源

場合によっては、ローカルのファイルだけでなく、外部の資料も文脈として使えます。

たとえば、

- API仕様書
- 公式ドキュメント
- 社内Wiki
- 外部のナレッジベース
- MCP でつないだドキュメント

などです。

これらも、エージェントにとっては「背景知識」として機能します。

---

## 5. 過去の作業履歴や判断材料

さらに、次のようなものも Context access に近い役割を持ちます。

- 過去の作業方針
- レビュー結果
- 「このファイルは触らない」といった注意
- 以前の会話から整理されたメモ

これは、単なるファイルの中身というより、**そのプロジェクトの歴史や判断の蓄積** です。

---

# Kindle本作成プロジェクトで何を置くと効くか

Kindle本作成のようなプロジェクトでは、Context access に置くものを  
**3つの層** に分けると管理しやすいです。

- 毎回読む基本方針
- 必要なときだけ使う専門手順
- 必要に応じて参照する資料

---

## 1. 毎回読む基本方針

これは `GEMINI.md` に置くのが向いています。

たとえば、以下のような内容です。

- この本の目的
- 想定読者
- 文体
- 避けたいこと
- 大事にしたい価値観

### 例

``` markdown
# GEMINI.md

## このプロジェクトの目的
ChatGPTやAIを使ったKindle本を、初心者にも読みやすく、実践しやすい形で作る。

## 想定読者
- AIに興味はあるが、技術者ではない人
- Kindle出版に興味がある個人
- パソコン作業はできるが、専門用語は多すぎない方がよい人

## 文体
- 親しみやすい
- 断定しすぎない
- 具体例を多くする
- 難語を使うときは一言で説明する

## 避けること
- 上から目線
- 抽象論だけで終わる説明
- いきなり専門用語を連発する書き方

## この本で重視すること
- すぐ試せる
- 失敗しやすい点も書く
- 著者の実体験を入れる
```
